2. Data Storage — Files, Databases, and Data Lakes

2.1 File Formats

FormatTypeProsCons

ParquetColumnarCompressed, fast analytics, schema embedded. Used in P053Not human-readable
ORCColumnarOptimized for Hive/Spark, ACID transactionsHive ecosystem lock-in

P053 data flow: Raw data generated as NumPy → saved as NPZ for training (2.1 GB) → streaming via Kafka as Avro/JSON → Spark ETL writes Parquet to S3 → model reads Parquet for batch inference and drift detection. 
# CSV vs Parquet comparison (actual P053 data)
import pandas as pd
import time

# CSV: 16M rows × 36 columns
t0 = time.time()
df = pd.read_csv('dram_16m.csv')  # 45 seconds, 8.2 GB RAM
print(f"CSV: {time.time()-t0:.1f}s, {df.memory_usage().sum()/1e9:.1f} GB")

# Parquet: Same data
t0 = time.time()
df = pd.read_parquet('dram_16m.parquet')  # 3.2 seconds, 1.8 GB RAM
print(f"Parquet: {time.time()-t0:.1f}s, {df.memory_usage().sum()/1e9:.1f} GB")

# CSV file: 4.5 GB on disk
# Parquet file: 890 MB on disk (5x smaller, Snappy compression)

3. Apache Kafka — Real-Time Data Streaming

Apache Kafka is a distributed event streaming platform. Think of it as a real-time message highway — producers send messages, consumers read them, and Kafka stores everything durably in between.

3.1 Architecture

ComponentRole
BrokerA Kafka server that stores and serves messages. Multiple brokers = a cluster.
TopicNamed channel for messages (like a table in SQL). P053: wafer-sensor-data
PartitionA topic is split into partitions for parallelism. Each partition is an ordered log.
ProducerSends messages to a topic. P053: Data generator pushes sensor readings
ConsumerReads messages from a topic. P053: Spark Streaming reads sensor data
Consumer GroupMultiple consumers sharing topic partitions. Load balancing.
OffsetPosition of a consumer in a partition. Consumers track where they left off.
Zookeeper / KRaftCluster coordination. KRaft (Kafka Raft) replaces Zookeeper in Kafka 3.x+
 
# Python Kafka producer (conceptual - P053 uses Confluent Kafka)
from confluent_kafka import Producer

producer = Producer({'bootstrap.servers': 'kafka:9092'})

# Send wafer sensor data
for wafer_id in range(5_000_000):
    data = generate_wafer_data(wafer_id)
    producer.produce(
        topic='wafer-sensor-data',
        key=str(wafer_id),
        value=json.dumps(data)
    )
producer.flush()

# Python Kafka consumer
from confluent_kafka import Consumer

consumer = Consumer({
    'bootstrap.servers': 'kafka:9092',
    'group.id': 'spark-etl-group',
    'auto.offset.reset': 'earliest'
})
consumer.subscribe(['wafer-sensor-data'])

while True:
    msg = consumer.poll(1.0)
    if msg is not None:
        process(json.loads(msg.value()))

3.2 When to Use Kafka (and When NOT to)

✅ Use Kafka When❌ Don't Use Kafka When
Real-time data needs (< 100ms latency)Simple batch processing (use Airflow + S3)
Multiple consumers need the same dataOne-time data migration
Decoupling producers from consumersData fits in a single database
Event sourcing / audit loggingSimple request-response (use REST API)
High throughput (millions of events/sec)Small dataset (< 10K records/day)

3.3 P053 Kafka Usage

P053 Pipeline: Kafka simulates a real semiconductor fab's data flow.
  • Topic: wafer-sensor-data — sensor readings from 33 process steps
  • Volume: 1M-10M messages/day (variable, simulating production variance)
  • Format: JSON messages with schema (wafer_id, features, timestamp, lot_id)
  • Consumer: Spark Structured Streaming reads batches every 5 minutes
  • Retention: 7 days (allows replay for debugging)

4. Apache Spark — Distributed Data Processing

Apache Spark is a distributed computing engine for large-scale data processing. It's 10-100× faster than Hadoop MapReduce because it processes data in-memory.

4.1 Architecture

ComponentRole
DriverYour main program. Creates SparkContext, builds execution plan, coordinates work.
Cluster ManagerAllocates resources (YARN, Mesos, K8s, or Standalone).
ExecutorWorker JVM on each node. Runs tasks, stores cached data.
TaskUnit of work on a data partition. Executor runs many tasks in parallel.
DAGDirected Acyclic Graph of operations. Spark builds this from your code.
StageGroup of tasks that can run without shuffling data.
ShuffleData redistribution across nodes (expensive — avoid when possible).

4.2 Spark SQL, DataFrames, and RDDs

from pyspark.sql import SparkSession
from pyspark.sql import functions as F

spark = SparkSession.builder \
    .appName("P053-ETL") \
    .config("spark.executor.memory", "4g") \
    .config("spark.sql.adaptive.enabled", "true") \
    .getOrCreate()

# Read Parquet (columnar, pushdown predicates)
df = spark.read.parquet("s3://p053-mlflow-artifacts/data/day_20/")

# Transformations (lazy — nothing executes yet)
processed = (
    df
    .filter(F.col("wafer_temp") > 25)         # Filter rows
    .withColumn("temp_leak_ratio",             # Feature engineering
                F.col("wafer_temp") / F.col("leakage_current"))
    .groupBy("lot_id")                         # Aggregation
    .agg(
        F.avg("yield_prediction").alias("avg_yield"),
        F.count("*").alias("wafer_count"),
        F.stddev("leakage_current").alias("leak_std")
    )
)

# Action (THIS triggers execution)
result = processed.collect()  # Brings data to driver
processed.write.parquet("s3://p053-mlflow-artifacts/data/day_20_processed/")

# SQL interface
df.createOrReplaceTempView("wafers")
spark.sql("""
    SELECT lot_id, AVG(yield_prediction) as avg_yield
    FROM wafers WHERE wafer_temp > 25
    GROUP BY lot_id HAVING COUNT(*) > 100
""").show()
Lazy evaluation: Spark doesn't execute anything until you call an action (.collect(), .count(), .write()). Transformations (.filter(), .withColumn()) just build the execution plan. This lets Spark optimize the entire pipeline before running it.

4.3 Spark MLlib — Distributed ML Training

from pyspark.ml.classification import LogisticRegression
from pyspark.ml.feature import VectorAssembler

# Assemble features into a single vector column
assembler = VectorAssembler(
    inputCols=["wafer_temp", "leakage_current", "timing_margin"],
    outputCol="features"
)

# Spark ML expects (features_vector, label) format
df_ml = assembler.transform(df).select("features", "label")

# Distributed logistic regression
lr = LogisticRegression(maxIter=100, regParam=0.01)
model = lr.fit(df_ml)
predictions = model.transform(df_ml)

# P053 note: We DON'T use Spark MLlib for the main model
# (PyTorch HybridTransformerCNN can't be distributed via Spark).
# Spark handles ETL; PyTorch handles training on a single GPU.

4.4 P053 Spark Usage

P053 Spark pipeline (in dag_daily_yield_pipeline.py):
  1. Ingest: Read from Kafka topic → Spark Structured Streaming
  2. ETL: Clean nulls, normalize features, compute engineered features
  3. Feature store: Write processed Parquet to S3
  4. Drift detection: Compute PSI, KL divergence, KS test on feature distributions
  5. Batch predict: Load PyTorch model → predict on processed data

Volume: 5M rows/day × 40 days = 200M total rows processed

5. Apache Airflow — Workflow Orchestration

Apache Airflow orchestrates complex data pipelines. It schedules tasks, manages dependencies, retries failures, and provides a visual UI.

5.1 Architecture

ComponentRole
SchedulerParses DAG files, creates task instances, assigns to workers. Runs continuously.
WebserverUI for monitoring DAGs, viewing logs, triggering runs
WorkerExecutes tasks (CeleryExecutor, KubernetesExecutor, or LocalExecutor)
Metadata DBPostgreSQL storing DAG state, task history, connections, variables
DAGDirected Acyclic Graph — defines tasks and their dependencies in Python
OperatorTemplate for a task: PythonOperator, BashOperator, SparkSubmitOperator, etc.
SensorWaits for an external condition (S3 file exists, API returns 200)
XComCross-communication — tasks share small data (< 48KB) between them

5.2 Writing DAGs (Python, not YAML)

from airflow import DAG
from airflow.operators.python import PythonOperator
from airflow.utils.dates import days_ago
from datetime import timedelta

# DAG definition
with DAG(
    dag_id='p053_daily_yield',
    schedule_interval='@daily',
    start_date=days_ago(1),
    default_args={
        'retries': 2,
        'retry_delay': timedelta(minutes=5),
        'email_on_failure': True,
    },
    catchup=False,
    max_active_runs=1,
    tags=['p053', 'production'],
) as dag:

    # Task 1: Generate data
    generate = PythonOperator(
        task_id='generate_daily_data',
        python_callable=generate_daily_data,
        op_kwargs={'rows': 5_000_000},
    )

    # Task 2: Spark ETL
    etl = PythonOperator(
        task_id='spark_etl',
        python_callable=run_spark_etl,
    )

    # Task 3: Drift detection
    drift = PythonOperator(
        task_id='detect_drift',
        python_callable=run_drift_detection,
    )

    # Task 4: Batch inference
    predict = PythonOperator(
        task_id='batch_predict',
        python_callable=run_batch_inference,
    )

    # Dependencies
    generate >> etl >> [drift, predict]
    #          etl ──> drift
    # generate ──> etl ──> predict

5.3 P053 Airflow DAGs — 3-DAG Architecture

P053 uses 3 DAGs on AWS g4dn.xlarge:
DAGScheduleTasks
dag_simulation_masterOnce (manual trigger)Orchestrates 40-day simulation, variable daily volumes, triggers daily DAG for each day, auto-stop EC2 at end
dag_daily_yield_pipelineTriggered per dayGenerate data → Kafka → Spark ETL → drift check → batch predict → log metrics → S3 upload
dag_retrain_pipelineTriggered by driftReal GPU training on T4 → canary test → promote/rollback in MLflow Registry → S3 upload

6.2 Model Registry

# Register model as a new version
model_uri = f"runs:/{run_id}/model"
mv = mlflow.register_model(model_uri, "P053-DRAM-Yield-Model")
# Creates version 1, 2, 3... automatically

# Set alias (like a pointer)
client = mlflow.tracking.MlflowClient()
client.set_registered_model_alias("P053-DRAM-Yield-Model", "champion", mv.version)

# Load production model
champion = mlflow.pyfunc.load_model("models:/P053-DRAM-Yield-Model@champion")

# Rollback (point alias to previous version)
client.set_registered_model_alias("P053-DRAM-Yield-Model", "champion", "2")
# Instant rollback — no redeployment needed

6.3 P053 MLflow Architecture

P053 MLflow stack:
  • Tracking server: MLflow container on EC2, connected to RDS PostgreSQL
  • Artifact store: S3 (s3://p053-mlflow-artifacts/mlflow/)
  • Model Registry: @champion and @challenger aliases for A/B testing
  • Retrain flow: Drift detected → train on T4 GPU → register new version → canary test → promote @champion
  • Rollback flow: Canary fails → revert @champion alias to previous version → log rollback event

8. Monitoring and Observability

8.1 Prometheus

Prometheus is a time-series metrics database. It pulls (scrapes) metrics from your services every 15 seconds.

# Python — expose metrics for Prometheus
from prometheus_client import Counter, Histogram, start_http_server

# Define metrics
PREDICTIONS = Counter('predictions_total', 'Total predictions', ['result'])
LATENCY = Histogram('prediction_latency_seconds', 'Prediction latency')

# In your API endpoint
@app.post("/predict")
async def predict(data):
    with LATENCY.time():
        result = model.predict(data)
    PREDICTIONS.labels(result='pass' if result > 0.5 else 'fail').inc()
    return {"prediction": result}

# Start metrics server on :8001
start_http_server(8001)

8.2 Grafana

Grafana visualizes Prometheus metrics in real-time dashboards.

P053 Grafana dashboards:
  • Inference dashboard: Request rate, latency (p50/p95/p99), error rate
  • Model performance: AUC-PR over time, prediction distribution, confidence scores
  • Drift monitoring: PSI per feature, KL divergence, KS test p-values
  • Infrastructure: GPU utilization, CPU usage, memory, disk I/O

12. Interview Questions with Answers

Q1: "What's the most complex data pipeline you've built?"

Answer: "P053 — a 40-day production simulation processing 200M rows of DRAM sensor data. Kafka streams 5M-10M rows/day to Spark for ETL and feature engineering. Airflow orchestrates 3 DAGs: a master simulation controller, a daily pipeline (Kafka→Spark→predict→drift), and a retrain pipeline that triggers real GPU training on T4 when drift exceeds PSI>0.10. All on a single g4dn.xlarge instance with auto-stop protection. MLflow tracks everything with PostgreSQL backend and S3 artifacts. The pipeline automatically retrained the model 3 times across 40 days and auto-stopped the EC2 instance at completion."

Q2: "How do you handle model drift in production?"

Answer: "Three-layer detection: PSI for distribution shift (>0.10 warning, >0.25 critical), KL divergence for information-theoretic drift, and KS test for statistical significance. When 3+ features cross critical thresholds AND model age > 7 days AND sufficient new data available (>50K rows), the retrain pipeline fires automatically. The new model goes through a canary test — we compare AUC-PR against the champion. If it improves, we promote via MLflow Model Registry alias swap. If not, we rollback the alias and log the failed attempt."

Q3: "Why Kafka instead of just batch processing?"

Answer: "In a real semiconductor fab, sensor data arrives continuously from test stations — not in neat daily batches. Kafka gives us: (1) Decoupling — the data generator doesn't need to know about Spark or the model. (2) Replay — if drift detection has a bug, we can reprocess last 7 days from Kafka. (3) Real-time capability — we can switch from 5-minute micro-batches to true streaming if latency requirements tighten. (4) Multiple consumers — Spark reads for ETL, a monitoring service reads for alerting, both from the same topic."

Q4: "Why Spark instead of Pandas/Polars?"

Answer: "I literally demonstrated pandas OOMing on 100M rows before introducing Spark. Pandas loads everything into RAM — on a g4dn.xlarge with 16GB, that's a 4-8GB limit. Our daily volume is 5-10M rows, and drift detection needs reference windows of 50M+ rows across multiple days. Spark processes this in partitions, spills to disk if needed, and scales horizontally if we add executors. Polars is faster than Pandas but still single-node — when your pipeline needs to grow to 100M+/day, Spark is the only answer."

Q5: "How do you decide between building vs buying infrastructure?"

Answer: "Cost-benefit analysis at every layer. We use OPEN SOURCE (Kafka, Spark, Airflow, MLflow, Prometheus, Grafana) instead of managed services (AWS Kinesis, EMR, Step Functions, SageMaker) because: (1) No vendor lock-in — the same pipeline runs on AWS, GCP, or on-prem. (2) Cost — managed Kafka (MSK) costs $300/month for a minimal cluster vs $0 for open source in Docker. (3) Learning — I can debug Kafka internals, not debug a black box. The tradeoff is operational overhead, which we mitigate with Docker Compose and ec2-user-data.sh bootstrap."
