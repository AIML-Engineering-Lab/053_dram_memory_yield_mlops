2.6   CloudWatch — Billing Alarm CODE READY, ACTIVATES ON EC2 LAUNCH

Coded in src/ec2_auto_stop.py — setup_billing_alarm(threshold_usd=200.0). This is called during EC2 bootstrap and creates a CloudWatch alarm in us-east-1 (billing metrics only exist in us-east-1) that SNS-notifies when monthly estimated charges exceed $200 USD.

At $0.526/hr (g4dn.xlarge) + $0.018/hr (RDS) = $0.544/hr combined, reaching $200 would require 368 hours of continuous runtime. Given our simulation runs <8 hours, the real risk is forgetting to stop instances — which the $200 alarm and ec2_auto_stop.py both guard against.

4   GPU Quota — Issue & Resolution
Current Status: Quota = 0 vCPUs for "Running On-Demand G and VT instances". Increase to 4 vCPUs requested via AWS Console (root account). Waiting for approval (15 min to 24 hrs). Once approved, EC2 launch proceeds with same command — no changes needed.

Why this happens: AWS defaults all new accounts to 0 GPU vCPUs as a fraud prevention measure. GPU instances cost $0.53–$32/hr, and without limits, a compromised account could accumulate thousands of dollars in minutes.

In enterprise: GPU quotas are pre-negotiated with AWS account teams during the annual Enterprise Discount Program (EDP) commitment. A company committing $1M/yr to AWS typically gets 100+ GPU vCPUs pre-approved. This is why ML teams at AMD/NVIDIA/Qualcomm don't hit this wall — their accounts are pre-configured.

Verification commands:

# Check quota status (may return AccessDenied if using IAM user — use root/console instead) aws service-quotas get-service-quota --service-code ec2 --quota-code L-DB2E81BA --region us-west-2 # Test with dry-run (works even without GPU quota) aws ec2 run-instances --image-id ami-0e1c254c6491f13f2 --instance-type t3.micro --key-name p053-key --security-group-ids sg-0f11ba29c1155cba3 --dry-run

The --dry-run with t3.micro returned "DryRunOperation" (would succeed) — confirming account is verified, only the GPU quota is missing.

6   Security Practices
  • No secrets in code: All credentials in .env.aws (gitignored). GitHub Actions uses GitHub Secrets for AWS keys.
  • IP-locked security group: All 6 ports restricted to your IP /32. Reduces attack surface vs 0.0.0.0/0.
  • IAM least privilege: p053-cicd-user has only EC2/S3/RDS/ECR/IAM policies — not root. CI/CD uses this user for ECR push.
  • No hardcoded credentials in Docker: EC2 instance profile (IAM role) used for S3/ECR access from EC2. No AWS keys on the instance.
  • RDS not public: RDS is in the same VPC as EC2. EC2 reaches it at private IP. RDS does not need a public endpoint — all traffic stays inside AWS.
  • Database password: Strong password, URL-encoded if it contains @ (replace @ with %40 in connection string). Stored in .env.aws (gitignored), not in source code.
  • .gitignore protects: deploy/aws/.env*, *.pem, __pycache__, mlruns/, mlartifacts/ all excluded from git.

7.3   MacBook Requirement
Short answer: NO — your MacBook does NOT need to stay on once the simulation is triggered.

When you launch the Airflow DAG on EC2:

  • All compute runs on EC2 g4dn.xlarge in AWS Oregon (us-west-2)
  • Airflow schedules and executes DAGs autonomously on the EC2 instance
  • GPU training runs inside EC2 (T4) — the subprocess call to train.py runs inside Docker, on EC2
  • MLflow writes to RDS (also on AWS) — no MacBook involvement
  • S3 artifact saves happen from EC2 to S3 — all within AWS network
  • ec2_auto_stop.py runs inside the Airflow DAG on EC2 — stops everything when done

You only need your MacBook for: triggering the DAG (one browser tab), and checking status (AWS Console mobile app or check_aws_services.sh). You can sleep, commute, or work on other things while the 2–3 hour simulation runs.

Monitoring from phone: Install "AWS Console" app (free, App Store). Shows EC2 status, RDS status, CloudWatch alarms, and S3. You'll get notified if the $200 billing alarm triggers.

7.4   RDS-Managed Elastic IPs

Your check_aws_services.sh will show an Elastic IP (34.214.51.57, eipalloc-0ae0823a57419a290) with ServiceManaged: rds. This is completely normal and safe — AWS RDS automatically allocates and manages this EIP for VPC connectivity. It is:

  • Created automatically when RDS was created — you did not allocate it
  • Managed by AWS — you cannot and should not release it manually
  • Released automatically when RDS is deleted
  • Not billed separately — included in RDS pricing

The check script now labels these as [AUTO-MANAGED by RDS] and suppresses the billing warning for them.
