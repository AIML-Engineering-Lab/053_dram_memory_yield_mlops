1. What is PyTorch? History, Philosophy, and Why It Won

PyTorch is an open-source machine learning framework developed by Meta AI (Facebook AI Research / FAIR). It was released in January 2017, built on top of the Torch library (Lua-based, developed at NYU). PyTorch provides two core capabilities:

  1. Tensor computation with GPU acceleration (like NumPy but on CUDA)
  2. Automatic differentiation (autograd) for building and training neural networks
Key philosophy: "Define-by-run" — the computation graph is built dynamically as you execute code. This means PyTorch code looks like regular Python. You can use if, for, while, print() freely. TensorFlow 1.x required you to define the entire graph before running it. This one decision won PyTorch the research community.

2. Tensors — The Foundation of Everything

A tensor is a multi-dimensional array — the data structure that flows through every neural network. Think of it as a NumPy ndarray that can live on a GPU.

DimensionsNameExample
0Scalartorch.tensor(3.14)
1Vectortorch.tensor([1, 2, 3]) — a feature vector
2Matrixtorch.randn(128, 36) — a batch of 128 samples × 36 features
33D Tensortorch.randn(32, 128, 36) — batch × sequence × features
44D Tensortorch.randn(32, 3, 224, 224) — batch × channels × H × W (images)

2.1 Creating Tensors (14 Ways)

import torch

# From data
x = torch.tensor([1.0, 2.0, 3.0])              # From list
x = torch.tensor([[1, 2], [3, 4]])               # 2D from nested list
x = torch.from_numpy(np_array)                    # From NumPy (shares memory!)

# Factory functions
x = torch.zeros(3, 4)                             # All zeros
x = torch.ones(3, 4)                              # All ones
x = torch.full((3, 4), fill_value=7.0)           # All sevens
x = torch.eye(4)                                  # Identity matrix
x = torch.arange(0, 10, 2)                        # [0, 2, 4, 6, 8]
x = torch.linspace(0, 1, 100)                     # 100 points from 0 to 1

# Random
x = torch.randn(128, 36)                          # Normal distribution (mean=0, std=1)
x = torch.rand(128, 36)                           # Uniform [0, 1)
x = torch.randint(0, 10, (128,))                  # Random integers

# Like another tensor
x = torch.zeros_like(other_tensor)                # Same shape/dtype/device
x = torch.randn_like(other_tensor)                # Random with same shape

# On specific device
x = torch.randn(128, 36, device='cuda')           # Directly on GPU
x = torch.randn(128, 36, device='mps')            # Apple Silicon
x = torch.randn(128, 36, dtype=torch.bfloat16)    # Specific dtype
Critical gotcha — torch.from_numpy() shares memory! If you modify the NumPy array, the tensor changes too (and vice versa). Use torch.tensor(np_array) to copy, or .clone() after.

2.2 Tensor Operations, Broadcasting, Indexing

# Element-wise operations
a + b                    # Addition (element-wise)
a * b                    # Multiplication (element-wise, NOT matrix multiply)
a @ b                    # Matrix multiplication (same as torch.matmul)
torch.matmul(a, b)       # Matrix multiplication (explicit)

# Reduction
x.sum()                  # Sum all elements → scalar
x.mean(dim=0)            # Mean along dimension 0 (batch mean)
x.max(dim=1)             # Max along dimension 1 (returns values, indices)
x.argmax(dim=1)          # Index of max along dimension 1

# Reshaping
x.view(32, -1)           # Reshape (must be contiguous)
x.reshape(32, -1)        # Reshape (always works, may copy)
x.permute(0, 2, 1)       # Swap dimensions (like numpy transpose)
x.unsqueeze(0)           # Add dimension at position 0
x.squeeze()              # Remove all dimensions of size 1

# Broadcasting (NumPy rules apply)
a = torch.randn(128, 36)   # (128, 36)
b = torch.randn(36)        # (36,) → broadcasts to (128, 36)
c = a + b                   # Works! b is broadcast along dim 0

# Indexing (Python-style)
x[0]                     # First row
x[:, 0]                  # First column
x[x > 0]                # Boolean indexing (elements > 0)
x[:, [0, 5, 10]]        # Fancy indexing (columns 0, 5, 10)

2.3 Device Management (CPU → GPU → MPS)

# Check availability
torch.cuda.is_available()       # True if NVIDIA GPU + CUDA
torch.backends.mps.is_available() # True if Apple Silicon

# Automatic device selection (P053 pattern)
def get_device():
    if torch.cuda.is_available():
        return torch.device('cuda')
    elif torch.backends.mps.is_available():
        return torch.device('mps')
    return torch.device('cpu')

device = get_device()

# Move tensors
x = x.to(device)                # Move to device
x = x.cuda()                    # Move to first CUDA GPU
x = x.cpu()                     # Move back to CPU

# Move model (moves ALL parameters)
model = model.to(device)

# CRITICAL: Data and model must be on SAME device
# This CRASHES: model on GPU, data on CPU
output = model(x_cpu)  # RuntimeError!
P053 pattern: We use get_device() at the top of train.py and pass it everywhere. This lets the same code run on T4 (CUDA), A100 (CUDA), M1 Mac (MPS), or CPU. Hardware-agnostic code is a principal-level requirement.

2.4 Memory Layout (contiguous, stride, view vs clone)

# Contiguous = elements stored sequentially in memory
x = torch.randn(3, 4)           # Contiguous
x_t = x.t()                      # Transpose — NOT contiguous!
x_t.is_contiguous()              # False

# .view() requires contiguous memory
x_t.view(-1)                     # RuntimeError!
x_t.contiguous().view(-1)        # Works after making contiguous
x_t.reshape(-1)                  # Always works (may copy internally)

# Stride: how many elements to skip to advance one position
x = torch.randn(3, 4)
x.stride()    # (4, 1) — skip 4 elements for next row, 1 for next column
x.t().stride() # (1, 4) — transposed strides, NOT contiguous

# clone() vs detach()
y = x.clone()        # New memory, keeps grad connection
y = x.detach()       # Same memory, breaks grad connection
y = x.clone().detach() # New memory + no grad — safest for metrics

3. Autograd — Automatic Differentiation Engine

Autograd is PyTorch's engine for computing gradients. Every tensor operation is recorded in a computation graph. When you call .backward(), autograd traverses this graph in reverse (backpropagation) to compute gradients.

3.1 Computation Graph (Dynamic vs Static)

# PyTorch builds the graph AS YOU EXECUTE operations
x = torch.tensor([2.0], requires_grad=True)
y = x ** 2 + 3 * x + 1      # Graph: x → x² → +3x → +1 → y
y.backward()                   # Compute dy/dx
print(x.grad)                  # tensor([7.0])  (2*2 + 3 = 7)

# DYNAMIC: The graph is rebuilt every forward pass
# This means you can use Python control flow!
def dynamic_model(x):
    if x.sum() > 0:             # Different path based on data
        return x * 2
    else:
        return x * 3
# TensorFlow 1.x couldn't do this without tf.cond!
Why "Dynamic"? The graph is built on-the-fly during the forward pass, not defined beforehand. This makes debugging trivial (print() works everywhere), and enables architectures with variable-length inputs, conditional branches, and loops. This is why 80% of ML research uses PyTorch.

3.2 Gradient Accumulation, Retention, and Zeroing

# CRITICAL: Gradients ACCUMULATE by default
optimizer.zero_grad()           # Always zero before backward!
loss.backward()                  # Compute gradients
optimizer.step()                 # Update weights

# Gradient accumulation (simulate larger batch)
# If GPU can only fit batch=128, but we want effective batch=512:
accumulation_steps = 4
for i, (x, y) in enumerate(dataloader):
    loss = model(x, y) / accumulation_steps  # Scale loss
    loss.backward()                           # Accumulate grads
    if (i + 1) % accumulation_steps == 0:
        optimizer.step()                       # Update
        optimizer.zero_grad()                  # Reset

# Gradient clipping (prevent exploding gradients)
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)

# Retain graph (for computing gradients twice)
loss.backward(retain_graph=True)  # Keep graph for second backward
Common bug: Forgetting optimizer.zero_grad(). Gradients accumulate across batches, causing the model to diverge. This is the #1 PyTorch beginner mistake. In P053, we zero gradients at the start of every training step.

4.4 P053 HybridTransformerCNN Architecture Deep Dive

Why Hybrid? DRAM yield has TWO signal types: (1) sensor interactions (30+ electrical measurements with physics-driven correlations) and (2) spatial patterns (die position → edge effect failures). A single model can't capture both efficiently.

# P053 Architecture (simplified from src/model.py)
class HybridTransformerCNN(nn.Module):
    """
    Transformer branch: 33 tabular features → per-feature embedding → 
                        self-attention → CLS token → 128d
    CNN branch:         3 spatial features → Conv1d(1→32→64) → 
                        GlobalAvgPool → FC → 64d
    Fusion:             concat(128 + 64) = 192d → MLP(192→128→64→1)
    """
    def __init__(self, n_tab=33, n_spatial=3, d_model=128, 
                 n_heads=4, n_layers=2, dropout=0.1):
        super().__init__()

        # Per-feature tokenization (FT-Transformer approach)
        # Each of 33 features gets its own Linear(1→128) embedding
        self.feature_embed = nn.ModuleList([
            nn.Linear(1, d_model) for _ in range(n_tab)
        ])

        # CLS token (learnable, like BERT)
        self.cls_token = nn.Parameter(torch.randn(1, 1, d_model))

        # Transformer encoder
        encoder_layer = nn.TransformerEncoderLayer(
            d_model=d_model, nhead=n_heads, dim_feedforward=256,
            dropout=dropout, activation='gelu', batch_first=True
        )
        self.transformer = nn.TransformerEncoder(
            encoder_layer, num_layers=n_layers
        )

        # CNN for spatial features
        self.spatial_cnn = nn.Sequential(
            nn.Conv1d(1, 32, kernel_size=1),
            nn.ReLU(),
            nn.Conv1d(32, 64, kernel_size=1),
            nn.ReLU(),
            nn.AdaptiveAvgPool1d(1)  # Global average pooling
        )
        self.spatial_fc = nn.Linear(64, 64)

        # Fusion MLP
        self.classifier = nn.Sequential(
            nn.Linear(d_model + 64, 128),  # 128 + 64 = 192
            nn.ReLU(),
            nn.Dropout(dropout),
            nn.Linear(128, 64),
            nn.ReLU(),
            nn.Linear(64, 1)               # Binary output (logit)
        )

    def forward(self, x_tab, x_spatial):
        # Tokenize each feature separately → (B, 33, 128)
        tokens = [emb(x_tab[:, i:i+1]) for i, emb in enumerate(self.feature_embed)]
        tokens = torch.stack(tokens, dim=1)  # (B, 33, 128)

        # Add CLS token → (B, 34, 128)
        cls = self.cls_token.expand(x_tab.size(0), -1, -1)
        tokens = torch.cat([cls, tokens], dim=1)

        # Transformer self-attention
        encoded = self.transformer(tokens)  # (B, 34, 128)
        cls_output = encoded[:, 0, :]       # CLS token → (B, 128)

        # CNN spatial branch
        spatial = x_spatial.unsqueeze(1)    # (B, 1, 3)
        spatial = self.spatial_cnn(spatial).squeeze(-1)  # (B, 64)
        spatial = self.spatial_fc(spatial)   # (B, 64)

        # Fusion → prediction
        fused = torch.cat([cls_output, spatial], dim=1)  # (B, 192)
        logit = self.classifier(fused)      # (B, 1)

        return logit.squeeze(-1)            # (B,)
Interview answer: "P053 uses per-feature tokenization — each of 33 sensor features gets its own 128-dimensional embedding. This lets the Transformer's self-attention discover pairwise interactions like temperature×leakage (Arrhenius coupling) without manual feature engineering. The CNN branch captures spatial patterns (edge die failures cluster in concentric rings). The Fusion MLP combines both for a 317K-param model that beats XGBoost."

# P053 FocalLoss implementation (from src/model.py)
class FocalLoss(nn.Module):
    def __init__(self, alpha=0.75, gamma=2.0, label_smoothing=0.01):
        super().__init__()
        self.alpha = alpha
        self.gamma = gamma
        self.label_smoothing = label_smoothing

    def forward(self, logits, targets):
        # Label smoothing: hard 0/1 → 0.01/0.99
        targets_smooth = targets * (1 - self.label_smoothing) + self.label_smoothing / 2
        bce = F.binary_cross_entropy_with_logits(logits, targets_smooth, reduction='none')
        pt = torch.exp(-bce)  # Probability of correct class
        focal_weight = (1 - pt) ** self.gamma  # Down-weight easy examples
        alpha_weight = targets * self.alpha + (1 - targets) * (1 - self.alpha)
        loss = alpha_weight * focal_weight * bce
        return loss.mean()

5.4 P053 Training Loop — Line by Line

# Simplified P053 training loop (from src/train.py)
model.train()
for epoch in range(num_epochs):
    running_loss = 0.0
    for batch_idx, (x_tab, x_spatial, labels) in enumerate(train_loader):
        # 1. Move data to GPU
        x_tab = x_tab.to(device)
        x_spatial = x_spatial.to(device)
        labels = labels.to(device)

        # 2. Zero gradients from previous step
        optimizer.zero_grad()

        # 3. Forward pass with AMP (mixed precision)
        with torch.amp.autocast('cuda', dtype=amp_dtype):
            logits = model(x_tab, x_spatial)
            loss = focal_loss(logits, labels)

        # 4. Backward pass
        if use_grad_scaler:   # float16 on T4
            scaler.scale(loss).backward()
            scaler.unscale_(optimizer)
            torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
            scaler.step(optimizer)
            scaler.update()
        else:                  # bfloat16 on A100 — NO scaler
            loss.backward()
            torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
            optimizer.step()

        running_loss += loss.item()

    # 5. Step LR scheduler
    scheduler.step()

    # 6. Validation (eval mode — disables Dropout, fixes BatchNorm)
    model.eval()
    with torch.no_grad():   # No gradient computation needed
        val_metrics = evaluate(model, val_loader, device, amp_dtype)
    model.train()            # Back to training mode

    # 7. Early stopping check
    if val_metrics['auc_pr'] > best_auc_pr:
        best_auc_pr = val_metrics['auc_pr']
        patience_counter = 0
        torch.save(model.state_dict(), 'best_model.pt')
    else:
        patience_counter += 1
        if patience_counter >= patience:
            print(f"Early stopping at epoch {epoch}")

7.3 GradScaler Deep Dive + P053 Death Spiral Bug

What GradScaler does:

  1. Multiplies loss by a large scale factor (e.g., 65536) before .backward()
  2. This amplifies small FP16 gradients so they don't underflow to zero
  3. After backward, divides gradients by the scale factor to get true values
  4. If NaN/Inf gradients detected → skips optimizer step, reduces scale by 0.5×
P053 Death Spiral (ED-031):
v2 (float16): GradScaler detected NaN at epoch 5 → halved scale → more NaN → halved again → scale reached 1.0 → underflow → all gradients zero → loss converged to constant 0.0693 (pure Focal Loss on all-negative prediction). Model brain-dead.

v3 (float16 + warmup): LR warmup delayed the collapse to epoch 7. Same death spiral.

v4 (bfloat16): 8-bit exponent handles the extreme gradient range. No GradScaler needed. Stable 50 epochs, AUC-ROC=0.816. This is the champion.

9. PyTorch 2.x — torch.compile and Beyond

9.1 TorchDynamo and TorchInductor

torch.compile (PyTorch 2.0+) is the biggest change since autograd. It captures Python code and compiles it into optimized GPU kernels.

  • TorchDynamo: Captures Python bytecode → builds FX graph (the "tracer")
  • TorchInductor: Compiles FX graph → optimized Triton/C++ kernels (the "compiler")

  • Typical speedup: 30-200% with zero code changes

9.3 Triton Integration (NVIDIA)

# TorchInductor generates Triton kernels automatically
# Triton = NVIDIA's Python-to-GPU compiler (easier than CUDA C++)

# You can also write custom Triton kernels
import triton
import triton.language as tl

@triton.jit
def fused_softmax_kernel(output_ptr, input_ptr, n_cols, BLOCK_SIZE: tl.constexpr):
    row_idx = tl.program_id(0)
    col_offsets = tl.arange(0, BLOCK_SIZE)
    mask = col_offsets < n_cols
    row = tl.load(input_ptr + row_idx * n_cols + col_offsets, mask=mask, other=-float('inf'))
    row_max = tl.max(row, axis=0)
    numerator = tl.exp(row - row_max)
    denominator = tl.sum(numerator, axis=0)
    tl.store(output_ptr + row_idx * n_cols + col_offsets, numerator / denominator, mask=mask)

10.1 Gradient Checkpointing (Memory vs Compute)

# Trade compute for memory — recompute activations during backward
from torch.utils.checkpoint import checkpoint

class MemoryEfficientModel(nn.Module):
    def forward(self, x):
        # Instead of storing all intermediate activations:
        x = checkpoint(self.layer1, x, use_reentrant=False)
        x = checkpoint(self.layer2, x, use_reentrant=False)
        return self.head(x)

# Saves ~50-70% memory at ~20-30% compute cost

13. Interview Questions with Answers

Q1: "What happens when you call loss.backward()?"

Answer: PyTorch traverses the computation graph in reverse (topological sort), computing gradients via the chain rule. Each tensor with requires_grad=True accumulates gradients in its .grad attribute. The graph is destroyed after backward (unless retain_graph=True). This is called "reverse-mode automatic differentiation" — it computes all partial derivatives in a single backward pass, which is O(n) for n parameters.

Q2: "Why bfloat16 instead of float16 on A100?"

Answer: bfloat16 has 8-bit exponent (same range as float32) vs float16's 5-bit exponent (max ~65K). In P053, FocalLoss with 1:160 imbalance generates extreme gradient values that overflow float16's range, causing GradScaler to repeatedly halve its scale factor until it reaches 1.0 — at which point gradients underflow to zero. bfloat16's wider range eliminates this entirely. Less precision in mantissa (7 vs 10 bits) doesn't matter for DNN training because SGD is inherently noisy.

Q3: "How would you scale this model to 1 billion parameters?"

Answer: Four changes: (1) FSDP or DeepSpeed ZeRO-3 to shard parameters across GPUs — a 1B model needs ~4GB in fp32, fits on 2× A100 with FSDP. (2) Gradient checkpointing to trade compute for memory — saves ~60% activation memory. (3) torch.compile for kernel fusion and reduced overhead. (4) Streaming DataLoader (IterableDataset) because dataset won't fit in RAM. Infrastructure: switch from g4dn.xlarge to p4d.24xlarge (8× A100 80GB).

Q4: "What's the difference between model.eval() and torch.no_grad()?"

Answer: model.eval() changes MODULE behavior: Dropout is disabled, BatchNorm uses running statistics instead of batch statistics. torch.no_grad() disables GRADIENT computation: no graph is built, saving memory and compute. For inference, you need BOTH: model.eval() for correct predictions, torch.no_grad() for efficiency. In P053, our validation loop uses both.

Q5: "How do you debug a model that produces NaN loss?"

Answer: Systematic approach: (1) torch.autograd.set_detect_anomaly(True) — this adds Python stack traces to the backward pass, showing exactly which operation produced NaN. (2) Check inputs: torch.isnan(x).any() before the model. (3) Check gradient norms after backward. (4) If using AMP, check GradScaler scale value — if it's dropping, your model is producing inf gradients. In P053, this is how we diagnosed the float16 collapse: GradScaler scale dropped from 65536 → 1.0 over 5 epochs.

Q6: "Explain your P053 architecture choice."

Answer: DRAM yield has two signal types requiring different inductive biases. Sensor features (33 electrical measurements) have complex pairwise interactions — temperature↔leakage follows Arrhenius physics, timing margins cascade across coupled paths. Self-attention in the Transformer branch discovers these interactions without manual feature engineering. Spatial features (3 wafer coordinates) have local structure — edge die failures cluster in concentric rings on the wafer. Conv1d captures this spatial locality. The Fusion MLP combines both 128d+64d representations. At 317K params, this outperforms XGBoost (0.054 vs 0.058 AUC-PR) while providing transfer learning capability for process node changes.

Q7: "How would you deploy this model on 100 AWS instances?"

Answer: Export to ONNX → TensorRT for inference optimization (2-5× speedup). Package in a Docker container with FastAPI endpoint. Deploy on ECS/EKS with HPA (horizontal pod autoscaler) scaling based on request latency. Model weights on S3, loaded at container startup. Blue-green deployment for zero-downtime updates. ALB (Application Load Balancer) for traffic routing. Prometheus + Grafana for latency/throughput monitoring. This is already designed in P053's Kubernetes manifests.
